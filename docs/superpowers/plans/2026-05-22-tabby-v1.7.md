# Tabby v1.7 — Account-aware auto-redirect Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a "strict-app" interception layer that, for any `*.google.com` URL containing a `/u/<digit>/` account segment, silently focuses an existing matching tab and closes the navigating tab — bypassing the duplicate and consolidate interstitials.

**Architecture:** A new pure-function ESM module `src/identity.mjs` exports `identityKey(url) → string | null`. `src/background.js` adds a `findAccountMatch` helper and a 4-step block at the top of `maybeIntercept` that runs before the existing duplicate and consolidate layers. No new settings; no new permissions.

**Tech Stack:** Vanilla ESM, Chrome MV3 service worker, `node:test` for unit tests.

**Spec:** `docs/superpowers/specs/2026-05-22-tabby-v1.7-design.md`

```js
/**
 * Returns a canonical "strict app instance" key for the URL, or null
 * if the URL doesn't match any strict rule.
 *
 * Today: matches *.google.com URLs with a /u/<digit>/ segment.
 * Future: this is the seam for additional rules (Notion workspaces,
 * Slack workspaces, etc.). Internals can grow into a rules table
 * without changing callers.
 */
export function identityKey(url) {
  let u;
  try {
    u = new URL(url);
  } catch {
    return null;
  }
  const host = u.hostname.toLowerCase();
  if (!host.endsWith(".google.com")) return null;
  const m = u.pathname.match(/^\/[^/]+\/u\/(\d+)\//);
  if (!m) return null;
  return `${host}|/u/${m[1]}/`;
}
```

- [ ] **Step 2.2: Add the `findAccountMatch` helper**

Find the existing `findSameHostTabs` function in `src/background.js` (around line 168). Directly after the end of that function (before `async function shouldOfferConsolidation`), add this new helper:

```js
/**
 * Find an existing tab whose URL has the same identity key as `key`,
 * excluding the navigating tab. Returns the most-recently-accessed
 * match, or null. Used by the strict-app layer in `maybeIntercept`
 * to silently focus an existing app+account tab.
 */
async function findAccountMatch(key, excludeTabId) {
  const tabs = await chrome.tabs.query({});
  const candidates = [];
  for (const t of tabs) {
    if (t.id === excludeTabId) continue;
    const u = tabUrlForPageMatch(t);
    if (!u || !isHttpUrl(u) || isExtensionUrl(u)) continue;
    if (identityKey(u) !== key) continue;
    candidates.push(t);
  }
  if (candidates.length === 0) return null;
  candidates.sort((a, b) => (b.lastAccessed ?? 0) - (a.lastAccessed ?? 0));
  return candidates[0];
}
```

- [ ] **Step 3.2: Insert the strict-app block**

Edit `src/background.js`. Replace this block:

```js

    if (t.url && isExtensionUrl(t.url)) return;

    const dupes = await findSamePageDuplicates(tabId, url);
```

With this block:

```js
    if (t.pendingUrl && isExtensionUrl(t.pendingUrl)) return;
    if (t.url && isExtensionUrl(t.url)) return;

    // Strict-app layer: for known multi-account apps (Google /u/N/),
    // silently focus an existing tab with the same host+account and
    // close the navigator. Runs before duplicate/consolidate layers.
    const key = identityKey(url);
    if (key) {
      const match = await findAccountMatch(key, tabId);
      if (match) {
        let focused = false;
        try {
          await chrome.windows.update(match.windowId, { focused: true });
          await chrome.tabs.update(match.id, { active: true });
          focused = true;
        } catch (e) {
          console.warn("Tabby strict-redirect: focus failed, falling through:", e);
        }
        if (focused) {
          try {
            await chrome.tabs.remove(tabId);
          } catch (e) {
            console.warn("Tabby strict-redirect: navigator close failed:", e);
          }
          return;
        }
        // focus failed → fall through to existing duplicate/consolidate layers
      }
    }

    const dupes = await findSamePageDuplicates(tabId, url);
```

Load the extension in `chrome://extensions` (Developer mode → "Load unpacked" or click "Reload" if already loaded). Then walk through these scenarios. Each one corresponds to a spec test case. If any fails, do **not** commit — diagnose and fix the underlying code, then re-run from Task 3.

**Scenario A — Cross-view auto-redirect (the headline case).**
1. Open `https://mail.google.com/mail/u/0/#inbox`.
2. In a brand-new tab, type `https://mail.google.com/mail/u/0/#sent` and press Enter.
3. **Expected:** the inbox tab activates (still showing `#inbox`); the new tab closes silently. No interstitial.

**Scenario B — Different account, no redirect.**
1. Open `https://calendar.google.com/calendar/u/0/r/week/2026/5/17`.
2. In a new tab, type `https://calendar.google.com/calendar/u/1/r/week/2026/5/17`.
3. **Expected:** no auto-redirect. The new tab loads the `/u/1/` calendar normally. (May trigger the consolidate interstitial if 3+ calendar.google.com tabs exist; that's expected.)

**Scenario C — No account ID, no redirect.**
1. Open `https://mail.google.com/mail/u/0/#inbox`.
2. In a new tab, type bare `https://mail.google.com`.
3. **Expected:** strict layer returns null; navigation proceeds. Google itself redirects to `/u/0/`, which **does** match — the new tab gets closed and the inbox tab activates. (This is acceptable, even desirable.)

**Scenario D — In-place navigation within an already-matching tab.**
1. From the Gmail `/u/0/#inbox` tab, click a link to another `/u/0/` view (e.g. an email).
2. **Expected:** no auto-redirect (the navigator tab is excluded from the match set). Navigation proceeds normally inside that tab.

**Scenario E — Multi-window match.**
1. Open Gmail `/u/0/#inbox` in window A.
2. In window B (a separate Chrome window), open a new tab and type `https://mail.google.com/mail/u/0/#sent`.
3. **Expected:** window A becomes focused; the inbox tab activates; the new tab in window B closes.

**Scenario F — Most-recently-accessed wins.**
1. Open two Gmail `/u/0/` tabs (`#inbox` and `#sent`).
2. Click into the `#sent` tab to make it most-recent.
3. In a third new tab, type `https://mail.google.com/mail/u/0/#drafts`.
4. **Expected:** the `#sent` tab activates (most recently accessed), not `#inbox`.

**Scenario G — Same-URL duplicate on a strict-app URL.**
1. Open `https://calendar.google.com/calendar/u/0/r/week/2026/5/17`.
2. In a new tab, type the **identical** URL.
3. **Expected:** strict layer wins before the exact-URL duplicate layer.
   The existing calendar tab activates; the new tab closes silently.
   No duplicate interstitial. (Strict-app match is a superset of
   exact-URL match within Google `/u/N/` apps.)

**Scenario H — Pinned existing tab.**
1. Open a Gmail `/u/0/` tab and pin it (right-click → Pin).
2. In a new tab, type `https://mail.google.com/mail/u/0/#sent`.
3. **Expected:** the pinned tab activates; the new tab closes. (Pinning
   has no special exclusion in `findAccountMatch`.)

**Scenario I — Pre-existing layers still work for non-strict URLs.**
1. Open `https://example.com/some/page`. Open a second tab to the exact same URL.
2. **Expected:** duplicate interstitial fires (unchanged v1.6 behavior).
3. Open three tabs to `https://www.ebay.com`.
4. **Expected:** consolidate interstitial fires on the third (unchanged v1.6 behavior, modulo cooldown).
