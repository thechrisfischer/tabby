# Tabby v1.6 — Configurable cooldown, growth re-arm, and host-level tab picker

**Status:** Design / not yet implemented
**Author:** chris@goodword.com (with Claude)
**Date:** 2026-05-17
**Supersedes parts of:** v1.4 consolidate-interstitial cooldown semantics

## Problem

The v1.4 consolidate interstitial fires once per host per six hours. In practice
that means the user gets warned the first time their ebay tabs cross the
three-tab threshold, then Tabby goes silent for the rest of the work session
even as the tab count climbs to ten or twenty. The cooldown was tuned to avoid
becoming a nag — instead it became invisible during the exact spurts of tab
sprawl Tabby is supposed to catch.

A second, related gap: when the consolidate interstitial does fire, it asks a
binary question ("open in this tab" or "tidy up and continue") without showing
the user which existing tabs are in play. The duplicate interstitial already
has the right interaction — a list of existing tabs the user can click to
focus — but it only triggers on exact URL matches.

## Goals

1. Replace the hard-coded six-hour cooldown with a user-configurable setting
   that ships with a sensible default (five minutes).
2. Add growth-aware re-arm: even inside a cooldown window, re-pitch when the
   user's tab count for a host has grown materially since the last pitch.
3. Bring the duplicate interstitial's "list existing tabs, click to focus"
   interaction to the consolidate interstitial, so users can recover their
   work instead of just closing or proceeding.

## Non-goals

- Per-host overrides ("never throttle ebay, always throttle linkedin"). Real
  need, but a v2 once we know how the default config feels.
- Cross-device settings sync (`chrome.storage.sync`). Cooldown is an
  opinionated focus setting; per-device is the right default.
- An options page. One knob fits in the popup; revisit when there are three or
  more.
- Changing the popup consolidation behavior (keep-2 + undo) shipped in v1.5.0.

## Design

### Part A — Configurable cooldown + smart re-arm

#### Settings UI

Add a `<select>` to `src/popup.html` directly below the existing consolidate
button, labeled **"Re-check sites every:"** with five options:

| Value (ms)    | Label                                |
|---------------|--------------------------------------|
| `0`           | Off — only when tabs keep stacking   |
| `300_000`     | 5 min                                |
| `1_800_000`   | 30 min                               |
| `3_600_000`   | 1 hour                               |
| `21_600_000`  | 6 hours                              |

Selection persists to `chrome.storage.local` under a single key,
`settings.consolidateCooldownMs`. Default value: `300_000` (5 minutes).

The popup loads the value on open, sets the `<select>`, and writes back on
change. No save button — change-on-blur semantics.

#### Storage shape

Single top-level key `settings` in `chrome.storage.local`:

```js
chrome.storage.local.set({
  settings: { consolidateCooldownMs: 300000 }
});
```

Reads use `chrome.storage.local.get("settings")`.

Background reads with a defensive fallback so missing/malformed settings cannot
brick the extension.

#### Pitch-record shape change

Today, `recordConsolidationPitch(host)` stores a single timestamp in
`chrome.storage.session`. v1.6 stores both timestamp **and** the tab count at
the moment of the pitch:

```json
{
  "consolidate_pitch_www.ebay.com": {
    "at": 1715980800000,
    "count": 4
  }
}
```

A bare number value is treated as legacy data: read as `{ at: <number>, count: 0 }`
so the growth check still works during the rollout window.

#### `shouldOfferConsolidation` logic

```
shouldOfferConsolidation(host, currentTabCount):
  pitch = storage.session.get(`consolidate_pitch_${host}`)
  if not pitch: return true                          # never pitched

  cooldownMs = settings.consolidateCooldownMs ?? 300000

  # cooldownMs === 0 means "Off — no time silence". The cooldown check is
  # disabled entirely; only growth can re-arm. Without this branch, 0ms would
  # elapse instantly and fire on every navigation — the opposite of the
  # intended "pitch the first time, then only on material change" semantic.
  cooldownElapsed = cooldownMs > 0 && (Date.now() - pitch.at) > cooldownMs

  grewSinceLastPitch = currentTabCount >= (pitch.count + REARM_GROWTH)

  return cooldownElapsed OR grewSinceLastPitch
```

`REARM_GROWTH` is a module-level constant defaulting to `2`. Tunable in code,
not in user settings — keeping the surface area small.

`maybeIntercept` already computes `tabCount` before this check, so we just pass
it through. `recordConsolidationPitch` is updated to accept `(host, count)`
and store both fields.

#### Behavior summary

| Setting | Time silence              | Growth re-arm                                     |
|---------|---------------------------|---------------------------------------------------|
| Off     | disabled — never re-arms  | sole re-arm path (+2 tabs since last pitch)       |
| 5 min   | 5 min                     | yes (+2 tabs) — fires sooner if growth hits first |
| 30 min  | 30 min                    | yes (+2 tabs)                                     |
| 1 hour  | 1 hour                    | yes (+2 tabs)                                     |
| 6 hours | 6 hours                   | yes (+2 tabs)                                     |

The two conditions are OR'd: whichever fires first re-arms the pitch. Even
"6 hours" still pitches on a runaway growth — users can't be silenced through
a tab explosion. "Off" is unique in that growth is the *only* re-arm path
after the first pitch.

### Part B — Host-level tab picker in the consolidate interstitial

#### Today's behavior

`showConsolidateInterstitial` stores `{ pendingUrl, hostname, tabCount, tabId }`
in session storage and navigates the tab to the interstitial. The interstitial
renders three buttons; it does not show which tabs exist on the host.

#### v1.6 behavior

Augment the consolidate payload with a `hostTabs` array shaped like the
existing `duplicates` array used by the duplicate interstitial — same fields,
collected via a new helper `findSameHostTabs(tabId, hostname)` modeled on
`findSamePageDuplicates`:

```typescript
type HostTab = {
  id: number;
  windowId: number;
  index: number;
  title: string;
  url: string;
  favIconUrl: string;
  lastAccessed: number;
  pinned: boolean;
};
```

Sort order: most-recently-accessed first (descending), then by window + index
as the secondary key. Excludes the navigating tab and any tab whose URL is not
http(s).

#### Interstitial UI

`src/interstitial.html` keeps the existing two-mode structure (`mode-duplicate`
and `mode-consolidate`) but the consolidate panel grows:

```
+------------------------------------------------------+
|  Quite a few tabs for this site                      |
|  www.ebay.com — 6 open                               |
|                                                      |
|  [favicon]  Vintage Polaroid SX-70 listing           |  <- existing tabs,
|             last seen 2 min ago             [Focus]  |     click row to focus
|  [favicon]  ebay Auctions: Cameras                   |     (Focus button is
|             last seen 14 min ago            [Focus]  |     the affordance;
|  [favicon]  …                                        |     whole row is also
|                                                      |     clickable)
|  ------------------------------------------------    |
|  [Open in this tab — Enter]                          |  <- existing actions
|  [Tidy up & continue — C]                            |     unchanged
|  [Stay on this screen — Esc]                         |
+------------------------------------------------------+
```

The existing `list` element from the duplicate panel is the visual reference;
we reuse its row styling. The consolidate panel gets its own `<ul id="con-list">`
populated by the same renderer logic, parameterized on which mode is active.

Click handler: posts `tabby-focus-tab` for the row's tab id (handler already
exists in background) and then posts `tabby-close-tab` for the **current
interstitial tab** (the tab the user typed ebay.com into — it's now showing
the interstitial and is the one we want to discard). End state: user lands on
the existing tab they picked; the redundant new-tab navigation is gone.

#### Action semantics — unchanged

The three existing buttons keep their current behavior:

- **Open in this tab** (`btn-keep`): clears bypass, lets navigation proceed.
- **Tidy up & continue** (`btn-merge`): sends `tabby-consolidate-host-and-go`,
  which closes all other http tabs on the host and proceeds. (Note: this is
  host-level "close all", different from the popup's keep-N. Worth a future
  unification, not in scope here.)
- **Stay on this screen** (`btn-con-stay`): closes the interstitial tab.

## Data flow

```
User types ebay.com (new blank tab)
   │
   ▼
chrome.webNavigation.onBeforeNavigate
   │
   ▼
maybeIntercept(tabId, "https://www.ebay.com")
   │
   ├─ findSamePageDuplicates  → none (existing tabs are product pages)
   │
   ├─ countTabsForHostnameAfterNavigation  → 6
   │
   ├─ shouldOfferConsolidation("www.ebay.com", 6)
   │     │
   │     ├─ read settings.consolidateCooldownMs
   │     ├─ read consolidate_pitch_www.ebay.com  → { at, count }
   │     └─ return cooldownElapsed OR grewSinceLastPitch
   │
   ├─ findSameHostTabs(tabId, "www.ebay.com")  → [HostTab, …]
   │
   └─ showConsolidateInterstitial(tabId, url, host, count, hostTabs)
         │
         └─ recordConsolidationPitch(host, count)
```

## Component changes

### `manifest.json`
- Bump `version` to `1.6.0`.
- No new permissions. `storage` is already declared.

### `src/background.js`
- New constant `REARM_GROWTH = 2`.
- New constant `DEFAULT_COOLDOWN_MS = 5 * 60 * 1000`.
- New helper `getSettings()` returning `{ consolidateCooldownMs }` from
  `chrome.storage.local`, with defensive fallbacks.
- `shouldOfferConsolidation(host, currentTabCount)` — signature change. Reads
  settings, applies cooldown OR growth check.
- `recordConsolidationPitch(host, count)` — signature change. Stores
  `{ at, count }`.
- New helper `findSameHostTabs(navigatingTabId, hostname)` mirroring the shape
  of `findSamePageDuplicates`.
- `showConsolidateInterstitial(tabId, pendingUrl, host, count, hostTabs)` —
  signature change, stores `hostTabs` in session payload.
- `maybeIntercept` — collects `hostTabs` before pitching; passes count into
  `shouldOfferConsolidation` and `recordConsolidationPitch`.
- Remove `CONSOLIDATE_COOLDOWN_MS` constant.

### `src/popup.html`
- Add `<select id="cooldown">` below the consolidate button with the five
  options above and the keeper-note paragraph.

### `src/popup.js`
- On load: read `settings.consolidateCooldownMs` from `chrome.storage.local`,
  set the select value, fall back to default if missing.
- On `change` of the select: write back to `chrome.storage.local`. No save UI.
- Keep all existing consolidate / undo behavior.

### `src/interstitial.html`
- Add `<ul id="con-list" class="list" role="listbox" hidden>` inside the
  consolidate panel, above the actions block.

### `src/interstitial.css`
- Reuse existing `.list` styles. May need a small tweak to the consolidate
  panel max-height / scroll behavior when the list is long.

### `src/interstitial.js`
- Parameterize the list renderer so it can populate either `#list` (duplicate
  mode) or `#con-list` (consolidate mode).
- In consolidate mode: render `hostTabs` rows, wire row click to focus + close
  the in-flight tab.

## Error handling

- Settings storage read errors: log to console, fall back to `DEFAULT_COOLDOWN_MS`.
  Never block consolidation.
- Pitch-record read errors: treat as "no prior pitch", show the interstitial.
  Conservative bias toward speaking up, not silence.
- `findSameHostTabs` errors: log, render the interstitial with an empty list
  (degrades to v1.5 behavior).
- `tabby-focus-tab` failure on row click: leave the interstitial in place, no
  loud error — the user can still use the action buttons.

## Testing

Manual repro plan after implementation:

1. Set cooldown to "Off". Open four ebay.com tabs in quick succession.
   Expected: consolidate interstitial fires on tab 3, then again on tab 5
   (3 → 5 is +2 growth).
2. Set cooldown to "5 min". Trigger the interstitial. Within 5 min, open one
   more tab on the same host (count +1). Expected: silent. Open another tab
   bringing count to +2. Expected: fires.
3. Set cooldown to "6 hours". Trigger the interstitial. Within seconds, open
   three more tabs. Expected: fires on the third (+2 growth wins over time
   silence).
4. Trigger the consolidate interstitial. Expected: list shows all other ebay
   tabs sorted most-recent first. Click a row → focus that tab, in-flight
   tab closes.
5. From the consolidate interstitial, click "Tidy up & continue". Expected:
   unchanged from v1.5 — all other host tabs close, navigation proceeds.
6. Settings persistence: change cooldown, close popup, reopen. Expected:
   selection persists. Restart Chrome. Expected: still persists
   (chrome.storage.local survives restart).

## Open questions

None blocking. The `REARM_GROWTH = 2` threshold is a guess; we'll tune after
real use.

## Rollback

v1.6 is additive over v1.5. To revert: drop the settings UI, remove
`getSettings`/`REARM_GROWTH`, restore the bare-number pitch shape, restore the
6-hour constant, remove the `con-list` element. No data migration needed —
v1.5 code ignores the new fields gracefully.
