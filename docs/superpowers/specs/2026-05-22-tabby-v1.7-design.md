# Tabby v1.7 — Account-aware auto-redirect for Google `/u/N/` apps

**Status:** Design / not yet implemented
**Author:** chris@goodword.com (with Claude)
**Date:** 2026-05-22
**Builds on:** v1.6 host-level consolidate interstitial

## Problem

Tabby has two interception layers today:

1. **Exact-URL duplicate** — full URL match (origin + path + sorted query + hash).
   Shows a tab-picker interstitial.
2. **Host-level consolidate** — 3+ tabs on the same host, throttled by cooldown.
   Shows a consolidate interstitial.

Both layers ask the user to make a choice. In multi-account Google apps
(Gmail, Calendar, Drive, Docs, …), the host alone is too coarse — `goodword`
and `skull-and-bones` tabs share `calendar.google.com` and get lumped into
the consolidate prompt together. And the exact-URL layer is too narrow —
`/u/0/r/week/...` and `/u/0/r/settings/...` are clearly the same account,
but their full URLs differ, so the dedup doesn't fire.

The user wants a stricter, automatic behavior for these apps: if an
existing tab is open to the *same host + same account ID*, silently focus
that tab — no interstitial, no choice.

## Goals

1. Introduce a new "strict-app" interception layer that runs **before**
   the duplicate and consolidate checks.
2. For Google `*.google.com` URLs containing `/u/<digit>/`, match on
   `host + account-id` as the identity key.
3. On match: focus the existing tab (most-recently-accessed), close the
   navigating tab. No interstitial.
4. Keep the seam clean enough that adding strict rules for other apps
   (Notion workspaces, Slack workspaces, etc.) is a single-file change
   later.

## Non-goals

- A rules table for multiple apps. YAGNI — ship the single Google rule
  behind a clean function signature; refactor to a table when the second
  rule arrives and we know what shape it needs.
- A user setting to disable the behavior. The whole point is automatic
  enforcement; if it proves wrong, we'll add a toggle then.
- A bypass affordance ("open this in a new tab anyway"). Users can still
  use a new window or incognito. Easy to add later if needed.
- Updating the focused tab's URL to match the user's typed URL. The
  intent is "go back to the screen that's already open," matching the
  duplicate interstitial's existing "go to existing tab" semantics.
- Generalizing to non-Google multi-account URL patterns in this release.

## Design

### Identity rule

A URL qualifies for strict matching when:

- Hostname (lowercased) ends in `.google.com`
- Path matches `^/[^/]+/u/(\d+)/` (e.g. `/mail/u/0/`, `/calendar/u/1/`,
  `/drive/u/0/`)

The **identity key** is `${hostname.toLowerCase()}|/u/${digit}/`. Same
host and same account-id produce the same key; different account-ids
produce different keys (so `/u/0/` and `/u/1/` tabs are distinct).

URLs without a `/u/<digit>/` segment (e.g. bare `mail.google.com`) return
`null` and fall through to the existing layers. We don't try to guess a
"default account" — if Google does its own server-side redirect to
`/u/0/`, that navigation will be intercepted in its own right.

### `src/identity.mjs` (new)

A small ESM module with one export:

```js
/**
 * Returns a canonical "strict app instance" key for the URL, or null
 * if the URL doesn't match any strict rule.
 *
 * Today: matches *.google.com URLs with a /u/<digit>/ segment.
 * Future: this is the seam for additional rules (Notion workspaces,
 * Slack workspaces, etc.). Internals can grow into a rules table
 * without changing callers.
 */
export function identityKey(url) {
  let u;
  try { u = new URL(url); } catch { return null; }
  const host = u.hostname.toLowerCase();
  if (!host.endsWith(".google.com")) return null;
  const m = u.pathname.match(/^\/[^/]+\/u\/(\d+)\//);
  if (!m) return null;
  return `${host}|/u/${m[1]}/`;
}
```

Pure function, no Chrome API dependencies, unit-testable in
`tests/identity.test.mjs`.

### `src/background.js` changes

#### New helper

```js
/**
 * Find an existing tab whose URL has the same identity key as `key`,
 * excluding the navigating tab. Returns the most-recently-accessed
 * match, or null.
 */
async function findAccountMatch(key, excludeTabId) {
  const tabs = await chrome.tabs.query({});
  const candidates = [];
  for (const t of tabs) {
    if (t.id === excludeTabId) continue;
    const u = tabUrlForPageMatch(t);
    if (!u || !isHttpUrl(u) || isExtensionUrl(u)) continue;
    if (identityKey(u) !== key) continue;
    candidates.push(t);
  }
  if (candidates.length === 0) return null;
  candidates.sort((a, b) => (b.lastAccessed ?? 0) - (a.lastAccessed ?? 0));
  return candidates[0];
}
```

Reuses the existing `tabUrlForPageMatch` (handles in-flight `pendingUrl`)
and the existing `isHttpUrl` / `isExtensionUrl` filters.

#### `maybeIntercept` layering

Add the strict layer at the top of the try block, after the bypass and
own-page checks, before the duplicate scan:

```js
async function maybeIntercept(tabId, url, _source) {
  if (!isHttpUrl(url) || isExtensionUrl(url)) return;
  const host = hostnameOf(url);
  if (await hasBypass(tabId, host)) return;

  // ... existing in-flight throttle and tab-get ...

  // NEW: strict-app match layer
  const key = identityKey(url);
  if (key) {
    const match = await findAccountMatch(key, tabId);
    if (match) {
      let focused = false;
      try {
        await chrome.windows.update(match.windowId, { focused: true });
        await chrome.tabs.update(match.id, { active: true });
        focused = true;
      } catch (e) {
        console.warn("Tabby strict-redirect: focus failed, falling through:", e);
      }
      if (focused) {
        try {
          await chrome.tabs.remove(tabId);
        } catch (e) {
          console.warn("Tabby strict-redirect: navigator close failed:", e);
        }
        return;
      }
      // focus failed → fall through to existing duplicate/consolidate layers
    }
  }

  // existing duplicate scan...
  // existing consolidate scan...
}
```

Two-step error handling is deliberate:

- **Focus failed** (target tab gone between query and update): fall
  through to the existing duplicate/consolidate layers, since the
  user hasn't been moved anywhere yet.
- **Focus succeeded, navigator close failed:** log and return. The
  user is on the right tab; the worst case is an orphan tab the user
  can close manually. Don't fall through — that would show an
  interstitial on top of an already-completed redirect.

### Module imports

`src/background.js` adds:

```js
import { identityKey } from "./identity.mjs";
```

`identity.mjs` exists as a sibling to the existing `cooldown.mjs`; both
are pure-function modules with co-located tests under `tests/`.

## Data flow

```
User types calendar.google.com/calendar/u/0/r/week/... (new tab or in-place)
   │
   ▼
chrome.webNavigation.onBeforeNavigate
   │
   ▼
maybeIntercept(tabId, url)
   │
   ├─ bypass / in-flight checks  (unchanged)
   │
   ├─ identityKey(url) → "calendar.google.com|/u/0/"
   │
   ├─ findAccountMatch(key, tabId)
   │     │
   │     ├─ scan all tabs (excluding navigator)
   │     ├─ filter by identityKey === key
   │     └─ pick most-recently-accessed
   │
   ├─ match found?
   │     │ yes:
   │     │   ├─ focus match.windowId
   │     │   ├─ activate match.id
   │     │   └─ remove tabId  ← return
   │     │
   │     │ no:
   │     ▼
   ├─ findSamePageDuplicates  (existing)
   ├─ consolidate logic       (existing)
```

## Component changes

### `manifest.json`
- Bump `version` to `1.7.0`.
- No new permissions.

### `src/identity.mjs` (new)
- Single export: `identityKey(url)`.
- Pure ESM module, no Chrome API dependencies.

### `src/background.js`
- New import: `identityKey` from `./identity.mjs`.
- New helper `findAccountMatch(key, excludeTabId)`.
- New layer in `maybeIntercept`, placed after bypass/in-flight checks
  and before `findSamePageDuplicates`.

### `tests/identity.test.mjs` (new)
- Unit tests mirroring `tests/cooldown.test.mjs` style.
- Positive cases:
  - `https://mail.google.com/mail/u/0/#inbox` → `mail.google.com|/u/0/`
  - `https://calendar.google.com/calendar/u/1/r/week/2026/5/25`
    → `calendar.google.com|/u/1/`
  - `https://drive.google.com/drive/u/0/my-drive`
    → `drive.google.com|/u/0/`
  - `https://docs.google.com/document/u/0/d/abc123/edit`
    → `docs.google.com|/u/0/`
- Negative cases:
  - `https://mail.google.com/` → `null` (no `/u/N/`)
  - `https://google.com/search?q=x` → `null` (no `/u/N/`)
  - `https://example.com/u/0/foo` → `null` (not `*.google.com`)
  - `https://mail.google.com/mail/u/abc/` → `null` (non-digit)
  - `not a url` → `null` (malformed)
  - `https://MAIL.GOOGLE.COM/mail/u/0/#inbox` → `mail.google.com|/u/0/`
    (case-insensitive host)

## Error handling

- **`identityKey` throws** — can't happen by design (try/catch on
  `new URL`, regex test on string). If it did, we'd log and treat the
  URL as non-qualifying.
- **`findAccountMatch` errors** — caught at the call site; log and
  fall through to existing layers.
- **`chrome.windows.update` / `chrome.tabs.update` failures** — target
  tab may have closed between query and update. Logged; we **do not**
  close the navigator (user has no destination), and we fall through
  to the existing duplicate/consolidate layers.
- **`chrome.tabs.remove` failure on navigator** — logged; no further
  action. Focus has already moved; an orphan navigator tab is an
  acceptable degradation.
- **Race with new tabs opening during the scan** — acceptable. The scan
  is best-effort; if a new matching tab appears mid-flight, the next
  navigation will catch it.

## Testing

### Unit
Run via `node --test tests/`. Covers `identityKey` exhaustively
(positives, negatives, case, malformed).

### Manual repro plan

1. **Cross-view auto-redirect.** Open `mail.google.com/mail/u/0/#inbox`.
   In a new tab, type `mail.google.com/mail/u/0/#sent`. Expected:
   existing tab focuses (still showing `#inbox`); new tab closes
   silently. No interstitial.

2. **Same-view exact-URL duplicate (regression).** Open
   `calendar.google.com/calendar/u/0/r/week/2026/5/17`. In a new tab,
   type the same URL. Expected: still falls through to v1.6 behavior —
   strict layer focuses the existing tab + closes navigator (since the
   strict match wins before the exact-URL duplicate layer runs). Same
   end state, no interstitial.

3. **Different account, no redirect.** Open
   `calendar.google.com/calendar/u/0/r/week/...`. In a new tab, type
   `calendar.google.com/calendar/u/1/r/week/...`. Expected: no strict
   match (different keys). Falls through to consolidate logic if 3+
   tabs on host; otherwise proceeds normally.

4. **No account ID, no redirect.** Open
   `mail.google.com/mail/u/0/#inbox`. In a new tab, type bare
   `mail.google.com`. Expected: strict layer returns null (no `/u/N/`);
   falls through. Google itself will redirect to `/u/0/`, which will
   re-enter `maybeIntercept` and *that* navigation will match and
   redirect.

5. **In-place navigation within an existing matching tab.** From the
   Gmail `/u/0/` tab itself, click a link to another `/u/0/` view.
   Expected: no redirect (the navigator tab is excluded from the match
   set); navigation proceeds normally.

6. **Multi-window match.** Have a Gmail `/u/0/` tab in window A.
   In window B, type `mail.google.com/mail/u/0/#sent`. Expected:
   focus jumps to window A (window focus + tab activate); window B
   navigator tab closes.

7. **Pinned existing tab.** Pin a Gmail `/u/0/` tab. New tab navigation
   to `/u/0/#sent`. Expected: pinned tab focuses; new tab closes.
   (Pinning has no special exclusion.)

8. **Most-recently-accessed wins.** Have two `/u/0/` tabs. Use one,
   then the other. Open a third with a `/u/0/` URL. Expected: the
   most-recently-used of the two existing tabs gets focus.

## Open questions

None blocking. One soft-tunable: **bypass affordance.** If users
complain about not being able to force two tabs to the same account,
we'll add a modifier-key bypass or a one-shot session bypass. Not in
scope.

## Rollback

v1.7 is additive over v1.6. To revert:

- Remove the `identityKey` import and the strict-layer block from
  `maybeIntercept`.
- Delete `src/identity.mjs` and `tests/identity.test.mjs`.
- Restore `version` to `1.6.0` in `manifest.json`.

No data migration. v1.6 code ignores the absence of the strict layer
cleanly — its existing duplicate and consolidate logic is unchanged.
